- **CLI file exploration design**: Avoid vague "CLI or shell" guidance for skill workflows -> make one path canonical. For wikify skills, wrap `ls`/`rg`/`cat` affordances inside the CLI so bundle reads are constrained, rendered, and logged; keep raw shell tools as debugging escape hatches.
- **Schema field renames sweep YAML and prompts**: Renaming `WriteRequest.evidence_v2 -> evidence` updated Python but missed `prompts/write.yaml`'s `input_schema` and `{evidence_v2}` placeholder. Grep `*.yaml` and `*.md` after every Pydantic field rename.
- **Bundle.open is strict; bootstrap constructs directly**: `Bundle.open(path)` requires `run/state.json`. Tests and helpers that just need a path-context dataclass construct `Bundle(root=...)` directly. The privileged bootstrap path is `run init`, which writes state.json then proceeds.
- **Metric without producer is a no-op**: Adding `M5_hit_rate` to eval is meaningless until `chunk_read` events are emitted. Ship the producer (`corpus find` / `corpus show`) in the same change as the consumer; wire an end-to-end test that proves overlap > 0.
- **Boundary loggers do not mutate the target pre-init**: For `run init` IO capture, tee to `tempfile.mkdtemp()` and promote files to `<bundle>/run/io/` only after init succeeds. A half-created `<b>/run/` is otherwise indistinguishable from a real bundle to layout detection.
- **No path-dependent qualifiers in active surface**: "v2 bundle", "post-pivot X", "atomic skill", "legacy fallback" describe history, not behavior. Live code/docs/skills should describe what something IS. Reserve historical framing for explicitly labelled history docs such as `tasks/skill-centric-redesign-plan.md`.
- **Drop dead duplication on rename**: When renaming `WriteEvidenceRefV2 -> WriteEvidenceRef`, the old `WriteEvidenceRef` (small form) was kept alive though no consumer read it. Rename + grep-for-callers + delete-orphan in the same change.
- **Don't ship dead commands in canonical docs**: `docs/filesystem-state-design.md` listed `wikify run set --corpus` and `corpus find --out/--in/...` flags that the actual CLI rejects. Doc examples must be derived from `--help` output, not aspiration.
- **Skill primitives must not smuggle workflows**: Bundle/setup/helper skills should expose unit operations and contracts, not bundled strategy choices like exploration order, readiness thresholds, or writer fanout. Workflow skills own those decisions.
- **MCP should not blindly mirror CLI verbs**: For Wikify, expose expressive corpus/wiki primitives plus resources, keep workflows in skills, and keep CLI as a first-class sibling adapter rather than describing it as legacy.
- **No meta-references in shipped code or docstrings**: Phrases like "per `tasks/mcp_plan.md`", "Phase 1 ships X", "in later phases", "see the plan" rot the moment the plan moves and contaminate the public surface with session-local framing. Source comments, docstrings, and skill docs must describe what the code IS and DOES. Historical motivation belongs in commit messages and explicitly labelled history docs only. Internal algorithm phases ("phase 1: cache lookup", "phase 2: bulk resolution") that describe code stages are fine.
