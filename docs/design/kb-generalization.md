# Wikify Knowledge Base Generalization

## Vision

Wikify evolves from an academic-paper pipeline into a general-purpose personal knowledge
base system. The user collects raw sources of any kind, Wikify enriches them with
semantic structure, and an LLM builds and maintains a curated wiki on top of that structure.
All three layers are separate, composable, and independently queryable.

## Three-Layer Architecture

```
+-------------------------------------------------------------+
|  Layer 3: Curated Wiki                data/wiki/            |
|  LLM-authored concept articles, discovered through epoch    |
|  passes over the corpus. Progressive: concepts emerge from  |
|  reading, stubs grow into full articles as evidence         |
|  accumulates. Self-linking. Maintained by the LLM.         |
+------------------+------------------------------------------+
                   | reads / cites
+------------------v------------------------------------------+
|  Layer 2: Enriched Index              data/ (existing)      |
|  ChromaDB embeddings, SQLite graph, topic tags, citation    |
|  graph, vibes, section summaries. Derived from raw.         |
|  Recomputable. Domain-agnostic.                             |
+------------------+------------------------------------------+
                   | ingests
+------------------v------------------------------------------+
|  Layer 1: Raw Sources                 data/raw/             |
|  Original files. Immutable. Any format: PDF, DOCX, PPTX,   |
|  HTML, Markdown, images, code. Never modified after ingest. |
+-------------------------------------------------------------+
```

**Key principle:** Each layer has one direction of dependency. The wiki reads from the enriched
index but never writes back to the raw layer. Outputs (papers, reports) are generated from
the wiki + index but remain separate unless explicitly promoted.

## Layer 1: Generalized Ingest

### Current state

`ingest/service.py` handles PDF, DOCX, PPTX. `store/models.py:Paper` has `doc_type` (`DocType`
enum: paper, report, proposal, note, presentation, web_article, markdown, wiki_article, image,
repo_readme, other) and `origin` (corpus vs generated).

### What changes (planned)

The following new ingester modules are planned but not yet implemented:

| Module | Handles | Key parsing step |
|--------|---------|-----------------|
| `ingest/markdown.py` | `.md`, `.txt` | Frontmatter extraction, heading-based chunking |
| `ingest/html.py` | `.html`, web clips | trafilatura or readability -> markdown, then markdown pipeline |
| `ingest/image.py` | `.png`, `.jpg`, `.webp` | alt-text / caption -> summary chunk |

`ingest/service.py:ingest_file()` would extend its format dispatch to these types. The rest
of the pipeline (chunking, embedding, topic extraction) already works on chunks and is
format-agnostic.

### Raw file conventions (planned)

```
data/raw/
  papers/          # existing PDFs
  web/             # HTML clips (Obsidian Web Clipper output, wget, etc.)
  notes/           # Markdown files (personal notes, READMEs)
  images/          # diagrams, screenshots
```

## Layer 2: Enriched Index (minimal changes needed)

The enriched layer is already domain-agnostic:
- ChromaDB embeddings work on any text chunks.
- The topic graph and citation graph degrade gracefully for non-academic sources (no DOIs,
  no formal citations -- but concept co-occurrence and hyperlink structure can substitute).
- `find_corpus_gaps`, `find_synthesis_opportunities`, and `search_papers` work on any corpus.

Planned additions:
1. **Hyperlink extraction** for web/markdown sources: extract `[text](url)` and `<a href>`
   as pseudo-citations into the `Citation` table with `source_type="hyperlink"`.
2. **Image indexing**: store alt-text + surrounding context as a single chunk with
   `chunk_type="image_context"`.
3. **`display_name()` generalization**: fall back to `domain + year + title_slug` for
   non-academic sources (e.g., `example.com 2024 - How to do X`).

## Layer 3: Curated Wiki (epoch model)

The wiki is a directory of LLM-authored Markdown files, built progressively from the enriched
index. The primary build mechanism is the **epoch model** -- autonomous concept discovery
over multiple passes. See `docs/design/wiki-wikipedia-model.md` for the full design spec.

### Storage

```
data/wiki/
  _index.md             # library catalog (generated by builder.py)
  _epoch.json           # epoch counter + convergence metrics [PLANNED]
  _unanswered.jsonl     # open questions from wiki query escalation
  domains/
    {domain}/
      _index.md         # domain master index
      _index_{theme}.md # per-theme index
      concepts/         # one article per concept
      themes/           # optional grouping themes
  syntheses/            # cross-domain synthesis articles
```

### Article format (epoch model)

Each article covers one named concept. The format follows `wiki-wikipedia-model.md`:

```markdown
---
concept: Atomic Layer Deposition
aliases: [ALD]
type: technique
related: [HfO2, CVD, conformality, self-limiting reaction]
importance: 0.87
status: full
epoch: 3
domain: material_science
---

## Definition
## Mechanism
## Key Facts
## In This Corpus
## Relationships
## Open Questions
```

This differs from the sitemap-pipeline article format (which uses three zones: Established,
Contested, Open). The epoch model format is concept-centric and Wikipedia-structured.
The three-zone format remains in use for `wiki init` (sitemap pipeline).

### `WikiArticle` SQLite model (already in `store/models.py`)

```python
class WikiArticle(SQLModel, table=True):
    id: str              # slug, e.g. "atomic_layer_deposition"
    title: str
    status: str          # stub | draft | full
    file_path: str       # relative path in data/wiki/
    source_ids: str      # JSON list of Paper.id values
    topic_keys: str      # JSON list of topic vocab keys
    domain: str
    created_at: datetime
    updated_at: datetime
    model: str
    needs_update: bool
```

### New SQLite models for epoch model (planned, in `store/models.py`)

```python
class ConceptRecord(SQLModel, table=True):
    id: str              # slugified name, PK
    name: str            # canonical display name
    aliases: str         # JSON list
    concept_type: str    # technique | material | phenomenon | method | theory | dataset
    domain: str
    importance: float    # 0-1, computed from concept graph
    epoch_discovered: int
    epoch_last_updated: int
    article_status: str  # none | stub | draft | full
    article_path: str

class ConceptRelation(SQLModel, table=True):
    id: int | None
    source_concept: str  # FK -> ConceptRecord.id
    target_concept: str  # FK -> ConceptRecord.id
    relation_type: str   # IS-A | PART-OF | USED-IN | ENABLES | CONTRASTS-WITH
    weight: float
    epoch: int

class EpochLog(SQLModel, table=True):
    id: int | None
    epoch: int
    triggered_by: str    # "user" | "ingest" | "schedule"
    started_at: datetime
    completed_at: datetime | None
    concepts_discovered: int
    stubs_upgraded: int
    articles_written: int
    contradictions_flagged: int
    cross_refs_added: int
    converged: bool
```

## Progressive Wiki Building

### Current modes (sitemap pipeline, implemented)

`wikify wiki init` -- bootstrap wiki using exploration agent + sitemap + mapreduce.
Writes theme and concept articles from a structured plan.

`wikify wiki expand [concept]` -- deepen one stub into a full article using
`map_chunks_to_topic` + `reduce_to_article`.

`wikify wiki sync` -- after new ingest, update stale articles (those with
`needs_update=True`) using new corpus evidence.

`wikify wiki audit` -- structural health report.

`wikify wiki health` -- orphan, staleness, synthesis gap report.

### Planned modes (epoch model)

`wikify wiki epoch` -- run one full epoch (concept discovery + graph + article writing
+ cross-ref + index rebuild). See `wiki-implementation-plan.md` for the module breakdown.

`wikify wiki epoch --until-convergence` -- run epochs until convergence criteria are met.

### Convergence

The wiki is considered converged when all three hold:
1. New concepts per epoch < 2% of total concept count
2. Stub ratio < 10%
3. No new contradictions flagged in the last epoch

## Q&A Modes

| Mode | What it queries | When to use |
|------|----------------|-------------|
| `--wiki-only` | `data/wiki/` articles only | Fast; curated knowledge |
| `--corpus-only` | Raw enriched index | Most current; direct evidence |
| `--hybrid` (default) | Wiki for context, corpus for evidence | Best quality |

The 5-level escalation protocol in `wiki query` (implemented in `cli.py`) reads from
increasingly deep sources before admitting a gap. Unanswered questions are appended to
`_unanswered.jsonl` and drive future wiki expansion.

## Output Promotion (planned)

Generated papers, reports, and Q&A outputs can be filed back to the wiki:

```
wikify promote data/output/review.md --as wiki_article --title "ALD Overview"
```

This would:
1. Convert the output to wiki article format (adds frontmatter, updates citations)
2. Add it to `data/wiki/syntheses/` or `data/wiki/queries/`
3. Create a `WikiArticle` row with `source_ids` derived from cited papers
4. Run the linker to backlink it from relevant concept articles

## Suggested Execution Order (full generalization)

1. **Epoch model** (`wiki/concepts.py`, `wiki/concept_graph.py`, `wiki/article.py`,
   `wiki/epoch.py`, CLI) -- primary remaining wiki work. See `wiki-implementation-plan.md`.
2. **Generalize ingest** (`MarkdownIngester`, `HTMLIngester`) -- unlock non-PDF corpora.
3. **`promote` command** -- close the loop: outputs become knowledge.
4. **Wiki Q&A MCP tools** -- expose wiki as a retrieval surface alongside the corpus.

## What This Looks Like in Practice

**Day 1**: Ingest 50 papers on ALD. Run `wiki epoch` -> concepts discovered, articles written.
Run `wiki epoch` again -> new concepts filled, stubs upgraded. Repeat until convergence.

**Week 1**: Add 20 more papers. Run `wikify ingest data/raw/ && wikify wiki epoch`
-> new concepts discovered, existing articles upgraded with new evidence or flagged for
contradiction review.

**On demand**: `wikify wiki query "what is the state of ALD for neuromorphic?"` ->
answer grounded in wiki articles, with escalation to raw corpus if wiki is insufficient.

**Monthly**: `wikify wiki audit` -> identifies orphaned articles, hub papers not yet
referenced in any article (graph drift), and split/merge candidates.
